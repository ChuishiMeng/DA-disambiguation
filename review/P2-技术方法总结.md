## 二、消歧技术路线全景

### 2.1 六大技术路线

| # | 技术路线 | 代表论文 | 核心思想 | 适用歧义类型 |
|---|---------|---------|---------|-------------|
| 1 | **两阶段消歧** | P01 (Disambiguate First) | 先消歧生成解释，再解析为 SQL | 语言歧义 |
| 2 | **多候选推荐** | P07 (Odin) | 生成候选 SQL 集 + Conformal Prediction 过滤 | Schema 歧义 |
| 3 | **交互式澄清** | P03 (AmbiSQL), P04 (EIG), P09 | 检测歧义 → 生成澄清问题 → 用户反馈 | 所有类型 |
| 4 | **知识注入** | P12, P13, P14 | 构建领域知识库，减少语义鸿沟 | 业务层歧义 |
| 5 | **Schema Linking** | P08 (LinkAlign) | 多轮检索 + Agent 辩论 + 精确解析 | 数据层歧义 |
| 6 | **用户画像** | P07 (Personalizer), P19 | 学习用户偏好，个性化消歧 | 用户层歧义 |

## 三、核心技术组件分析

### 3.1 歧义检测

| 方法 | 技术 | 准确率 | 来源 |
|------|------|--------|------|
| LLM 0-shot 分类 | Prompt + 分类头 | ~70% | P03 |
| 训练分类器 | 8类分类 (PRACTIQ) | 未报告 | P05 |
| SBERT 相似度 | 多候选差异检测 | 启发式 | P07 |
| LLM 反思 | 多 Agent 辩论 | ~87% (P03) | P03, P08 |

**结论**: 歧义检测是消歧的前提，目前最佳方法结合 LLM 分类 + 后验证。

### 3.2 澄清问题生成

| 方法 | 策略 | 优化目标 | 来源 |
|------|------|---------|------|
| 模板生成 | 基于歧义类型选择模板 | 覆盖率 | P03 |
| 期望信息增益 (EIG) | 最大化信息增益 | 最少轮次 | P04 |
| 检索式 | 从 QA 库检索区分性问题 | 区分度 | P10 |
| LLM 生成 | Prompt 直接生成 | 自然度 | 多篇 |

**结论**: P04 的 EIG 策略理论上最优（最小化交互轮次），P10 的检索式最稳定。

### 3.3 Schema Linking 消歧

| 方法 | 策略 | EM (Spider) | 来源 |
|------|------|-------------|------|
| DIN-SQL | 单次 Prompt | 38.6% | P08对比 |
| MCS-SQL | 两步链接 | 29.1% | P08对比 |
| RSL-SQL | 双向链接 | 33.1% | P08对比 |
| **LinkAlign (Agent)** | 多轮检索 + Agent 辩论 | **47.7%** | P08 |
| **Odin Generator** | Schema Masking | N/A | P07 |

**结论**: Schema Linking 错误 > 60% 是主要瓶颈（P08分析），多轮+Agent 方法显著优于单次调用。

### 3.4 用户偏好学习

| 方法 | 反馈类型 | 学习方式 | 来源 |
|------|---------|---------|------|
| Odin Personalizer | 显式（选择SQL） | 文本提示 + SBERT微调 | P07 |
| 隐式用户画像 | 隐式（对话行为） | 对话行为分析 | P19 |
| 情境感知画像 | 混合 | 动态更新 | P18 |

**结论**: 显式+隐式结合是最佳方案：冷启动用隐式，迭代用显式。

## 五、现有方法局限性总结

### 5.1 按层面的局限性

| 层面 | 现有覆盖 | 关键空白 |
|------|---------|---------|
| **用户层** | P07 (显式), P19 (隐式) | ❌ 无统一用户画像驱动的消歧框架 |
| **业务层** | P12, P13, P14 (知识注入) | ❌ 知识注入未专门针对消歧优化 |
| **数据层** | P07, P08 (较好) | ⚠️ 未与业务层/用户层协同 |

### 5.2 跨层面空白

1. **三层协同缺失**: 各方法独立解决单一层面，无统一框架
2. **歧义分类不统一**: AMBROSIA(3类) vs PRACTIQ(8类) vs AmbiQT(3类)，缺乏统一标准
3. **评估标准分散**: 不同论文使用不同指标和数据集，难以直接比较
4. **真实场景不足**: 大部分评估基于人工修改的基准（Spider变体），非真实企业场景
5. **用户画像冷启动**: Odin 需要显式反馈，冷启动问题未解决

## 六、技术选型建议

### 6.1 我们框架的核心技术栈

| 组件 | 推荐技术 | 理由 |
|------|---------|------|
| **歧义检测** | LLM 分类 + Schema差异检测 | 综合P03和P07的优势 |
| **澄清问题** | EIG + 模板混合 | P04的理论 + P03的工程化 |
| **Schema Linking** | LinkAlign Pipeline模式 | 平衡效率与精度 |
| **用户偏好** | 隐式(冷启动) + 显式(迭代) | P19 + P07结合 |
| **知识注入** | 概念定义库 + 规则库 | P13的构建方法 + P12的动态适配 |

| 机会 | 新颖性 | 证据 |
|------|--------|------|
